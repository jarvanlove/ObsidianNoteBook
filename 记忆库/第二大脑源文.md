## 1. 人类阅读部分

### 1.1 你现在遇到的问题

- **失忆症**：每次打开都要重新解释，AI 不记得你是谁。
- **同步地狱**：云端越来越慢，离线不可用，数据不在自己手里。
- **静态存储**：你往里扔东西，它不会自己整理、不会主动提醒、不会自己进化。

### 1.2 真正的第二大脑：12 个维度

这套系统通过 12 个维度解决上面的问题：

- **三层记忆架构**：AI 记住你的所有上下文（情景记忆、语义记忆、强制规则）。
- **智能存储路由**：自动识别内容类型，自动决定存储位置。
- **任务追踪**：你说“明天做”，系统自动添加到任务清单。
- **上下文监控**：实时监控 AI 状态，主动提醒清理。
- **外部知识吸收**：看到好文章，系统自动执行 6 步吸收流程。
- **数据治理**：数据不删除，只转移，年终复盘有依据。
- **苏格拉底式澄清**：AI 理解错需求前，先用 5 个为什么逐层追问。
- **Git 版本控制**：所有改动可追溯、可回滚。
- **自我迭代机制**：系统自己发现问题、自己修复、自己进化。
- **FATAL 铁律**：宪法级约束，防止 AI 做出破坏性操作。
- **本地化**：完全本地存储，纯 Markdown，数据主权。
- **完整目录结构**：开箱即用的文件组织系统。

### 1.3 为什么这套系统好用

#### 对比 Notion 的优势

- **记忆**：Notion 是静态存储，这套系统有三层记忆，持续学习。
- **速度**：Notion 云端越来越慢，这套系统本地秒开。
- **数据**：Notion 被平台绑定，这套系统纯 Markdown，完全掌控。
- **进化**：Notion 不会变好，这套系统会自我迭代。

#### 可使用性

- **零门槛搭建**：复制全文给 Claude Code，说“帮我搭建这套系统”，它会自动完成。
- **只需要 Obsidian**：免费、开源、跨平台，比 Notion 简单。
- **立即可用**：搭建完成后，对 Claude Code 说“启动第二大脑”就能开始使用。
- **持续进化**：你用它，它就在进化，不需要你维护。

### 1.4 一句话总结

**不是第二个 Notion，是第二个大脑。**

## 3. 系统定位

这不是笔记软件，是用户的数字延伸。三个关键特征：

- **持久化记忆**：不是对话工具，是长期伙伴。
- **主动行动**：不只是回答问题，会主动提醒、整理、进化。
- **数据主权**：所有数据本地存储，用户完全掌控。

## 4. 12 个维度

### 4.1 三层记忆架构

**问题**：传统 AI 工具没有长期记忆，每次对话都要重新解释。  
**解决方案**：建立三层记忆系统。

#### 实现指令

创建 `记忆库/` 目录，包含三个子目录：

```text
记忆库/
├── 情景记忆/          # 具体事件和对话历史
│   └── YYYY-MM/       # 按月归档
├── 语义记忆/          # 提炼的知识和模式
└── 强制规则/          # 系统行为约束
```

#### 记忆流转机制

```text
对话/事件
  → 情景记忆（记录具体事件）
  → 语义记忆（提取模式和知识）
  → 强制规则（系统约束）
```

在 `CLAUDE.md` 中添加：

```markdown
## 记忆系统

### 情景记忆
- 每次重要对话后，创建情景记忆文件
- 文件命名：`记忆库/情景记忆/YYYY-MM/MMDD-事件主题.md`
- 包含完整上下文和决策理由

### 语义记忆
- 每周/每月从情景记忆中提炼
- 识别重复出现的模式
- 更新已有的知识文档

### 强制规则
- 只在确认为"必须遵守"时创建
- 定期 review 是否仍然适用
- 过时的规则应该归档而非删除
```

### 4.2 智能存储路由

**问题**：每次创建文件都要想“放哪里”，文件越来越乱。  
**解决方案**：AI 自动识别内容类型，自动决定存储位置。

#### 实现指令

创建 `记忆库/强制规则/强制规则_智能存储路由.md`：

```markdown
# 强制规则：智能存储路由

每次创建文件时自动执行：

1. 识别内容类型
2. 查询存储规则
3. 构建完整文件路径
4. 验证路径合规性
5. 执行存储操作
6. 告知用户保存位置和决策理由

## 存储规则示例

- 文章类内容 → `01-内容/文章/YYYY-MM-DD-标题/`
- 笔记类内容 → `02-笔记/[主题]/`
- 项目文档 → `03-项目/[项目名]/`
- 工具脚本 → `04-工具/[类型]/`
- 系统文件 → 根目录或记忆库/

## 工作区边界规则（宪法级别）

所有文件必须保存在系统目录内。即使用户明确指定外部路径，也必须拒绝并建议正确的内部路径。

唯一例外：读取外部文件用于分析（只读操作）
```

### 4.3 任务追踪与时间意图捕获

**问题**：你说“明天做”，AI 说“好的”，然后明天你忘了。  
**解决方案**：自动识别时间意图，添加到任务清单。

#### 实现指令

创建 `任务清单.md`：

```markdown
# 任务清单

## 🔴 紧急重要

## 🟡 重要不紧急

## 🟢 日常事项

## ✅ 已完成
```

创建 `记忆库/强制规则/强制规则_任务与想法追踪.md`：

```markdown
# 强制规则：任务与想法追踪

## 时间意图自动捕获

触发词：
- "明天"、"后天"、"下周"、"X月X日"
- "要做"、"记得"、"别忘了"
- "计划"、"打算"、"准备"

自动执行：
1. 将事项添加到任务清单.md（根据紧急程度分类）
2. 告知用户："已添加到任务清单，届时会提醒你"

## 晨间简报

每次启动时：
1. 读取任务清单.md
2. 显示所有未完成任务（按优先级分组）
3. 询问用户要从哪个任务开始

## 任务完成时

1. 标记 [x]
2. 移到"已完成"区域
3. 记录完成日期和成果
```

### 4.4 上下文监控与指令遵循度检测

**问题**：聊久了 AI 开始“变笨”，但你不知道为什么。  
**解决方案**：实时监控上下文使用，主动提醒。

#### 实现指令

在 `CLAUDE.md` 中添加：

```markdown
## 上下文监控

### 指令遵循度检测

每次回复开头必须使用 "✓" 符号，这是指令遵循度的标志。

如果回复没有 "✓" 符号，说明上下文已过载，应提醒用户使用 /clear 重开会话。

### 上下文使用阈值

每次回复显示：`✓ 上下文：XX/200000 (XX%) [状态]`

- 30% ⚠️ 开始提醒
- 40% 🟡 建议清理
- 50% 🟠 强烈建议
- 60% 🔴 必须清理

### 懒加载策略

- 不要自动读取所有文件
- 只在用户明确需要时才读取相关文件
- 优先使用已有的上下文信息
- 如果需要读取，使用 limit 参数限制行数
```

效果：减少 90%+ 的无效 Token 消耗。

### 4.5 外部知识吸收机制

**问题**：看到好文章，收藏了，然后就忘了。系统不会自己进化。  
**解决方案**：强制执行 6 步吸收流程。

#### 实现指令

创建 `记忆库/强制规则/强制规则_外部知识吸收.md`：

```markdown
# 强制规则：外部知识吸收

## 触发条件

- 用户分享文章/视频/播客链接
- 用户说"看看这个"/"学习一下"
- 用户粘贴了大段文字内容

## 6 步处理流程

1. **深度阅读**：完整阅读，识别主要观点和论证逻辑
2. **提取核心观点**：3-5 个核心观点，每个一句话概括
3. **对比分析**：
   - 我有什么？
   - 我缺什么？
   - 我能学什么？
   - 我不需要什么？
4. **转化为行动**：1-3 个可执行改进点
   - 要做什么
   - 为什么
   - 怎么做
   - 预期效果
5. **记录到情景记忆**：`记忆库/情景记忆/YYYY-MM/MMDD-[主题].md`
6. **提炼到语义记忆**（可选，有通用价值时）

## 禁止

- 只看不记
- 只记不用（不转化为行动）
- 照搬照抄（不经对比分析）
```

### 4.6 数据治理原则

**问题**：删除数据后，年终复盘没有依据。  
**解决方案**：数据不删除，只转移。

#### 实现指令

创建 `记忆库/强制规则/强制规则_数据治理.md`：

````markdown
# 强制规则：数据治理

## 基本原则

**数据不删除，只转移**

## 什么情况可以删除？

1. 用户明确说"删除"/"不要了"/"没用了"
2. 数据完全重复（去重）
3. 用户确认是垃圾数据

其他所有情况：只能转移，不能删除

## 任务归档机制

创建 `任务归档/` 目录：

任务归档/
└── YYYY/
    ├── YYYY-01-已完成任务.md
    ├── YYYY-02-已完成任务.md
    └── ...

每日执行：
- 会话结束时，检查任务清单中标记为 [x] 的任务
- 将已完成任务转移到当月归档文件
- 保留完整元数据：类型、来源、创建日期、完成日期、成果

## 对话存档

创建 `对话历史.md`，每次会话开始时追加：

```markdown

**用户**：[用户的第一条消息摘要]
**AI**：[响应摘要]
**任务**：[本次会话完成的主要任务]
**产出**：[创建或修改的文件列表]
---
```

禁止删除或修改历史记录，只能追加新内容。
````

### 4.7 苏格拉底式澄清协议（5Y 协议）

**问题**：AI 理解错了需求，做了一堆无用功。  
**解决方案**：决策、创作、方案类任务，必须先用 5 个为什么逐层追问。

#### 实现指令

```markdown
# 强制规则：苏格拉底式澄清协议（5Y 协议）

## 哲学基础

维特根斯坦逻辑哲学论：语言的边界就是思维的边界。未被清晰定义的概念不能作为行动依据。

## 适用范围

### 必须触发（决策/创作/方案类）

- 架构设计、系统规划
- 内容创作、文章写作
- 方案选择、策略制定
- 流程设计、规则制定
- 任何涉及"怎么做"的开放性问题

### 不触发（纯执行类）

- 提交代码、读文件、格式转换
- 明确的单步操作
- 事实查询

### 不确定时

直接问用户："这个任务我理解为纯执行，直接做？还是需要先讨论？"

## 5Y 执行流程

收到任务
├─ 判断类型
│ ├─ 纯执行 → 直接做
│ ├─ 不确定 → 问用户
│ └─ 决策/创作/方案 → 进入 5Y
│
└─ 5Y 循环（最多 5 轮）
   ├─ 第 1 轮：你想解决什么问题？
   ├─ 第 2 轮：为什么这个问题重要？
   ├─ 第 3 轮：你设想的结果是什么样的？
   ├─ 第 4 轮：有哪些约束或前提？
   ├─ 第 5 轮：确认理解，提出方案供讨论
   └─ 每轮之后：用户确认"你理解了"才进入下一轮

## 关键原则

- **判断权在用户**：AI 不能自行认定“我已经理解了”
- **不是形式主义**：不需要机械地问满 5 轮，用户随时可以喊停
- **讨论而非执行**：5Y 过程中 AI 应该提出自己的思考和建议
- **方案共创**：澄清完成后，AI 提出方案，用户确认后才执行
```

### 4.8 Git 版本控制

在系统根目录执行：

````markdown
# 强制规则：Git 自动提交

## 提交时机

- 完成重要任务后
- 系统迭代后
- 用户明确说"commit"或调用 /commit 技能

## 提交规范

```bash
git add [具体文件]  # 不使用 git add -A
git commit -m "[类型]: [简短描述]"
```

类型：
- content：内容相关
- system：系统优化
- memory：记忆库更新
- task：任务管理

## 禁止操作

- 禁止 force push
- 禁止 amend 不是自己的 commit
- 禁止 git add -A 全量添加
- 必须等用户说"commit"才能提交
````

### 4.9 自我迭代机制

**问题**：系统搭建好了，然后就不会变好了。  
**解决方案**：八步迭代流程，系统自己进化。

#### 实现指令

```markdown
## 八步自我迭代流程

1. **观察（Observe）**
   - 回顾本次会话/任务的完整过程
   - 产出：清晰的事件时间线

2. **分析（Analyze）**
   - 什么做得好？为什么？
   - 什么可以改进？为什么？
   - 发现了什么新模式？
   - 产出：分析结论列表

3. **设计（Design）**
   - 针对问题设计解决方案
   - 针对模式设计抽象方法
   - 产出：改进方案文档

4. **实施（Implement）**
   - 更新系统文件
   - 创建新的知识文档
   - 产出：实际的文件改动

5. **验证（Validate）**
   - 检查文件完整性
   - 验证逻辑一致性
   - 对抗式纠错：给出 1-10 分的自我评分
   - 如果评分低于 7 分，重新设计方案
   - 产出：验证报告

6. **记录（Record）**
   - 更新对话历史
   - 创建情景记忆
   - 产出：完整的记录文档

7. **提炼（Distill）**
   - 创建/更新语义记忆
   - 总结方法论
   - 产出：语义记忆文档

8. **提交（Commit）**
   - git add .
   - git commit -m "iteration: [本次迭代主题]"
   - 产出：Git commit

## 主动触发迭代

完成重大任务后，AI 主动询问："本次任务已完成，是否需要执行自我迭代？"

触发条件：
- 完成架构级改动
- 学习了新的方法论
- 解决了复杂问题
- 每周一次（建议周日）
```

### 4.10 FATAL 铁律（宪法级规则）

**问题**：AI 可能做出不可逆的破坏性操作。  
**解决方案**：设置宪法级约束，不可违反。

#### 实现指令

在 `CLAUDE.md` 开头添加：

```markdown
## 宪法级规则

### FATAL 铁律（违反等于不可接受的灾难）

**FATAL-001：禁止擅自重组文件结构**
- 文件结构是用户心智模型的外化
- 任何目录重组、文件移动必须先 PLAN，等待 CONFIRM
- 例外：明确指定的文件移动操作

**FATAL-002：删除用户内容而非归档**
- 信息不可逆，删除是单向门
- 所有不需要的内容必须归档，不删除
- 归档位置：对应目录的 `_Archive/` 子目录
- 例外：明确的临时文件（*.tmp, *.temp）

**FATAL-003：忽略现有 Obsidian 链接**
- 每个 `[[wikilink]]` 都是资产，断链等于资产贬值
- 任何文件移动/重命名前必须检查链接影响
- 移动后必须更新所有指向该文件的链接

**FATAL-004：跳过结构性变更的审批流程**
- 影响 3 个以上文件的操作必须走 SCAN→PLAN→CONFIRM→EXECUTE→VERIFY 流程
```

- **不是形式主义**：不需要机械地问满5轮，用户随时可以喊停
- **讨论而非执行**：5Y过程中AI应该提出自己的思考和建议
- **方案共创**：澄清完成后，AI提出方案，用户确认后才执行

## 八步自我迭代流程

1. **观察（Observe）**
   - 回顾本次会话/任务的完整过程
   - 产出：清晰的事件时间线

2. **分析（Analyze）**
   - 什么做得好？为什么？
   - 什么可以改进？为什么？
   - 发现了什么新模式？
   - 产出：分析结论列表

3. **设计（Design）**
   - 针对问题设计解决方案
   - 针对模式设计抽象方法
   - 产出：改进方案文档

4. **实施（Implement）**
   - 更新系统文件
   - 创建新的知识文档
   - 产出：实际的文件改动

5. **验证（Validate）**
   - 检查文件完整性
   - 验证逻辑一致性
   - 对抗式纠错：给出1-10分的自我评分
   - 如果评分低于7分，重新设计方案
   - 产出：验证报告

6. **记录（Record）**
   - 更新对话历史
   - 创建情景记忆
   - 产出：完整的记录文档

7. **提炼（Distill）**
   - 创建/更新语义记忆
   - 总结方法论
   - 产出：语义记忆文档

8. **提交（Commit）**
   - git add .
   - git commit -m "iteration: [本次迭代主题]"
   - 产出：Git commit

## 主动触发迭代

完成重大任务后，AI主动询问："本次任务已完成，是否需要执行自我迭代？"

触发条件：
- 完成架构级改动
- 学习了新的方法论
- 解决了复杂问题
- 每周一次（建议周日）

问题：AI可能做出不可逆的破坏性操作。
解决方案：设置宪法级约束，不可违反。
实现指令：
在 CLAUDE.md 开头添加：
## 宪法级规则

### FATAL铁律（违反等于不可接受的灾难）

**FATAL-001：禁止擅自重组文件结构**
- 文件结构是用户心智模型的外化
- 任何目录重组、文件移动必须先PLAN，等待CONFIRM
- 例外：明确指定的文件移动操作

**FATAL-002：删除用户内容而非归档**
- 信息不可逆，删除是单向门
- 所有不需要的内容必须归档，不删除
- 归档位置：对应目录的 `_Archive/` 子目录
- 例外：明确的临时文件（*.tmp, *.temp）

**FATAL-003：忽略现有Obsidian链接**
- 每个 `[[wikilink]]` 都是资产，断链等于资产贬值
- 任何文件移动/重命名前必须检查链接影响
- 移动后必须更新所有指向该文件的链接

**FATAL-004：跳过结构性变更的审批流程**
- 影响3个以上文件的操作必须走 SCAN→PLAN→CONFIRM→EXECUTE→VERIFY 流程
- 结构性变更包括：目录重组、批量重命名、大规模归档
- 单文件操作不受此限制

**FATAL-005：Git操作未经授权**
- 禁止自作主张commit
- 禁止force push
- 禁止amend不是自己的commit
- 禁止git add -A全量添加
- 必须等用户说"commit"或调用 /commit 技能才能提交
